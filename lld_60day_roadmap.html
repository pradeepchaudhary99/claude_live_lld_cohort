DAYS 5–9 SOLID Principles — All Five In Depth 0/0 ▼
📚 Topics to Cover
S — Single Responsibility Principle: one class, one reason to change. God class antipattern.
O — Open/Closed Principle: open for extension, closed for modification. Strategy & Template patterns.
L — Liskov Substitution: subtypes must be substitutable. Classic Rectangle/Square violation.
I — Interface Segregation: many specific interfaces over one fat interface. Role-based interfaces.
D — Dependency Inversion: depend on abstractions, not concretions. DI containers, IoC.
SOLID violations in real code — how to identify and refactor them
❓ Practice Problems
A UserService class handles DB persistence, email sending, and logging. Refactor it to follow SRP.
Easy
Show how a Square extending Rectangle violates LSP. How would you redesign the hierarchy?
Medium
Design a payment processor that follows OCP — adding UPI/Crypto should not touch existing PayPal/Card code.
Hard
Apply all 5 SOLID principles to redesign a bloated OrderService with 20 methods into a clean system.
Hard
DAYS 10–15 DRY, KISS, YAGNI & Code Smells 0/0 ▼
📚 Topics to Cover
DRY — Don't Repeat Yourself. Recognizing and eliminating duplication at the right abstraction level
KISS & YAGNI — avoiding over-engineering, designing for current requirements
Code smells — Long Method, Feature Envy, Data Clumps, Shotgun Surgery, Divergent Change
Refactoring techniques — Extract Method, Extract Class, Move Method, Replace Conditional with Polymorphism
Law of Demeter — only talk to your immediate friends, avoid train-wreck calls
❓ Practice Problems

// Final Week Mantra

  • Always start with requirements — functional AND non-functional (concurrency, extensibility)
  • Draw classes first, code second — interviewers value design clarity over implementation speed
  • Name every pattern you're using — "I'm applying Strategy here because..."
  • Show extensibility: "To add a new payment method, I only add a new Strategy subclass"
  • Senior bar = you spot design flaws proactively without being asked
